# 성경 암송 — 회원 버전(사용자 식별) 기획·실행 안 v1

> 작성일: 2026년 6월
> 대상: 고척교회 제자양육부 신앙운동팀 (내부용)
> 관계: 기존 익명 버전(`bible-memorize-app`)은 **그대로 유지**, 본 문서는 사용자 식별 기능을 가진 **별도 버전** 신설 안

## 0. 핵심 결정 요약 (확정)

| 항목 | 결정 |
|---|---|
| 진행 데이터 저장 | **로컬 우선 + 서버 백업** (localStorage 즉시 반영 + Google Sheets 누적) |
| 코드/사이트 분리 | **새 GitHub 저장소** (`bible-memorize-app-members`, 독립 URL) |
| 사용자 식별 입력 | **기존 summer-bible 등록 화면 구조 재사용** — 구분(교구/교회학교) 2분기. 1회 입력 후 기본값으로 채워 **확인 후 진행** |

> 식별 구조는 기존 [summer-bible/registration.html](https://github.com/sewoongkim1/summer-bible/blob/main/registration.html)의 입력 체계를 그대로 차용해 운영팀·성도에게 익숙한 분류를 쓴다.

## 1. 목표와 범위

### 1-1. 목표
- 기존 익명 암송 테스트에 **교구·목장·이름 식별**을 더해, 개인별로 진행 상태를 저장하고 운영팀이 **교구·목장·개인별 진행 현황을 집계·관리**할 수 있게 한다.
- 암송 기능(빈칸 채점·음성 암송)은 기존 버전을 **그대로 복제·재사용**한다. 새로 만드는 것은 ① 진입(식별) 화면, ② 서버 저장 연동, ③ 관리(집계)뿐이다.

### 1-2. 기존 버전과의 관계

| 구분 | 익명 버전 (기존, 유지) | 회원 버전 (신규) |
|---|---|---|
| 식별 | 없음 (완전 익명) | 교구 + 목장 + 이름 |
| 진행 저장 | localStorage만 | localStorage(우선) + Google Sheets(백업·집계) |
| 저장소 | `bible-memorize-app` | `bible-memorize-app-members` (신규) |
| 말씀 데이터 | 공용 시트/`verses.json` 재사용 가능 | **동일 말씀 데이터 재사용** |
| URL | 현재 사이트 | 별도 GitHub Pages URL |

> 두 버전은 말씀 마스터 데이터(시트1)는 공유하되, **진행 기록과 코드베이스는 분리**한다.

## 2. 시스템 아키텍처

기존과 동일한 무료 스택을 새 저장소로 복제한다.

```
[신규 저장소: bible-memorize-app-members]  ── GitHub Pages (독립 URL)
   │
   ├─ 말씀 데이터 읽기(GET)
   │     1순위: verses.json (기존 버전 것 재사용 / 또는 동일 Actions로 갱신)
   │     2순위: Apps Script doGet  (기존 그대로)
   │
   └─ 진행 기록 쓰기(POST)
         Apps Script doPost  → Google Sheets '진행기록' 시트에 append   ★신규
```

### 2-1. Apps Script 확장
- 기존 `doGet()`(말씀 조회)은 **수정 없이 유지**.
- `doPost(e)` **신규 추가**: 요청 본문(JSON)을 받아 '진행기록' 시트에 한 줄 append 후 `{ok:true}` 반환.
- **CORS 주의**: Apps Script Web App은 프리플라이트(OPTIONS)를 처리하지 못하므로, 프런트에서 `Content-Type: text/plain`(또는 `fetch ... mode:"no-cors"`)으로 보내고 `doPost`에서 `e.postData.contents`를 `JSON.parse` 한다. (응답 확인이 필요하면 text/plain 방식 권장)

### 2-2. 새 Google Sheets — '진행기록' 시트

교구·교회학교 두 분기를 공통 열에 담는다. (소속 = 교구명 또는 부서명, 세부 = 목장 또는 학년)

| 열 | 헤더 | 설명 | 교구 예시 | 교회학교 예시 |
|---|---|---|---|---|
| A | 일시 | 서버 기록 시각 | 2026-06-25 14:03 | 2026-06-25 14:03 |
| B | 구분 | 교구 / 교회학교 | 교구 | 교회학교 |
| C | 소속 | 교구명 또는 부서명 | 사랑 | 초등부 |
| D | 세부 | 목장 또는 학년 | 3 | 3학년 |
| E | 직분 | (교구만, 선택) | 집사님 | — |
| F | 성명 | 이름 | 김성도 | 김믿음 |
| G | 구절No | 테스트한 구절 번호 | 7 | 7 |
| H | 단계 | 통과 단계(1~3) | 2 | 2 |
| I | 방식 | typing / voice | typing | voice |
| J | 클라이언트ID | 기기 구분용 익명 UUID(선택) | a1b2c3 | a1b2c3 |

> 운영팀은 이 시트를 **필터·피벗**으로 구분별/소속별(교구·부서)/개인별 진행률을 집계한다. (별도 관리자 페이지는 2차 과제 — 6장 참고)

## 2-3. 사용자 식별 구조 (기존 등록 화면 재사용)

진입 화면은 먼저 **구분(교구 / 교회학교)** 을 라디오로 고르고, 선택에 따라 하위 입력이 바뀐다. (registration.html의 `onTypeChange()` 분기와 동일)

### 구분 1) 교구
| 필드 | 입력 방식 | 값 |
|---|---|---|
| 교구 | 라디오 | 믿음 · 소망 · 사랑 · 섬김 · 은혜 · 화평 · 기쁨 · 새가족 (8) |
| 목장 | 입력 | 숫자 (예: 3) |
| 직분 | 라디오(선택) | 성도님 · 집사님 · 안수집사님 · 권사님 · 장로님 · 목사님 · 사모님 |
| 성명 | 입력 | 이름 |

### 구분 2) 교회학교
| 필드 | 입력 방식 | 값 |
|---|---|---|
| 부서 | 라디오 | 사랑부 · 영아부 · 유아부 · 유치부 · 유년부 · 초등부 · 중등부 · 고등부 · 청년부 (9) |
| 학년 | 입력 | 예: 3학년 |
| 성명 | 입력 | 이름 |

> 직분은 등록 화면과 동일하게 **선택 항목**으로 둔다. 암송 진행 관리에 필수는 아니므로 미입력 허용.

## 3. 화면 흐름 (User Flow)

```
[진입 화면 — 식별 입력]   ★신규
  ├─ 첫 방문: 구분(교구/교회학교) 선택 → 분기별 입력
  │      교구    → 교구(라디오)·목장(입력)·직분(선택)·성명(입력)
  │      교회학교 → 부서(라디오)·학년(입력)·성명(입력)
  │      → [시작]
  └─ 재방문: 저장된 정보를 기본값으로 채워 보여줌
             → "사랑교구 3목장 김성도 님으로 계속" / "초등부 김믿음 님으로 계속"
                [확인] / [정보 변경]
       ↓ 확인
[목록 화면]  (기존 재사용)
  └─ 26개 구절 카드 + 진행 배지(미시도/1·2단계/암송완료)
       ↓ 카드 클릭
[테스트 화면]  (기존 재사용 — 빈칸 채점 + 음성 암송)
       ↓ 단계 통과 / 음성 90% 통과
[저장]
  ├─ localStorage 즉시 갱신 (배지 반영)
  └─ Apps Script doPost로 '진행기록' 시트에 백업  ★신규
```

## 4. 데이터 설계 (클라이언트)

### 4-1. 식별 정보 (localStorage)
```
key:   "memorize-user"

// 교구
value: { type:"교구", gu:"사랑", mok:"3", jik:"집사님", name:"김성도", cid:"a1b2c3" }
// 교회학교
value: { type:"교회학교", bu:"초등부", grade:"3학년", name:"김믿음", cid:"a1b2c3" }
        - cid: 최초 1회 생성하는 익명 기기 식별 UUID (중복 집계 보정용, 선택)
```
- 진입 화면에서 이 값이 있으면 입력칸의 **기본값으로 채우고 '확인 후 진행'**, 없으면 신규 입력.
- '정보 변경'을 누르면 다시 입력받아 덮어쓴다.

### 4-2. 진행 상태 (localStorage — 기존 구조 재사용)
```
key:   "memorize-progress"
value: { "1": { stage: 2, passed: true }, ... }
```
- 로컬이 **표시의 1차 출처**(즉시 반영, 오프라인 동작).

### 4-3. 서버 저장 페이로드 (doPost 전송)
```json
// 교구
{ "type":"교구", "gu":"사랑", "mok":"3", "jik":"집사님", "name":"김성도",
  "no":7, "stage":2, "mode":"typing", "cid":"a1b2c3" }
// 교회학교
{ "type":"교회학교", "bu":"초등부", "grade":"3학년", "name":"김믿음",
  "no":7, "stage":2, "mode":"typing", "cid":"a1b2c3" }
```
- 단계 통과/음성 통과 시 호출. 실패해도 로컬 진행은 유지(낙관적 처리), 다음 통과 때 재시도.

## 5. 개인정보 보호 (회원 버전용 별도 원칙)

- 수집 항목은 **교구·목장·이름**과 진행 기록(구절No·단계)으로 **최소화**. 연락처·주민번호 등 민감정보는 일절 받지 않는다.
- 내부 운영·격려 목적으로만 사용하며, 외부 공개·타 시스템 연동은 하지 않는다.
- 진입 화면에 "교구·목장·이름은 암송 진행 관리를 위해 내부적으로만 사용됩니다" 안내 문구 표기.
- 기존 익명 버전과 데이터·코드가 분리되어 서로 연결되지 않는다.
