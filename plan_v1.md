## 1. 프로젝트 개요

### 1-1. 배경

기존 "말씀 한 절 암송 소책자"(26주, 시편 119:105 ~ 히브리서 3:1)는 종이 기반으로 따라 쓰기와 자가진단(正자 획수 표시)을 통해 암송 여부를 스스로 점검하는 방식이었다. 이를 보완하여, 성도가 스마트폰이나 PC로 언제든 자신의 암송 상태를 직접 테스트하고 확인할 수 있는 온라인 도구를 도입한다.

### 1-2. 목표

- 시중 암송 앱(예: YouVersion 암송 기능 등)처럼 빈칸 채우기 → 점차 난이도 상승 → 전체 타이핑의 단계형 테스트 제공
- 입력한 답과 정답을 비교해 정확도를 즉시 보여주는 자가 채점 기능
- 기존 "완주 등록 시스템"과는 독립적으로 운영 (교구·목장·성명 식별 체계를 공유하지 않음)
- Google Sheets를 말씀 데이터 저장소로 재사용하여 운영팀이 손쉽게 구절을 추가·수정

### 1-3. 기존 시스템과의 관계

| 구분 | 완주 등록 시스템 (기존) | 암송 테스트 앱 (신규) |
|---|---|---|
| 목적 | 필사·퀴즈 완주 여부 등록 | 개인 암송 숙련도 자가 점검 |
| 식별 방식 | 교구+목장+성명 (중복 검사용) | 불필요 (개인 기기 내 또는 익명 세션) |
| 데이터 저장 | Google Sheets (전체 제출 현황) | Google Sheets (말씀 마스터 데이터, 별도 문서) |
| 연동 여부 | — | 연동 없음 (독립 시스템) |

> 💡 두 시스템은 "고척교회 성경 프로그램"이라는 같은 우산 아래 있지만, 데이터베이스·식별자·운영 로직은 분리해서 설계한다. 추후 필요 시 통합은 가능하지만 현재 범위에는 포함하지 않는다.

## 2. 테스트 방식 설계

시중 암송 앱들이 공통적으로 쓰는 "점진적 난이도 상승" 패턴을 차용한다. 사용자가 처음에는 일부 단어만 채우고, 익숙해지면 빈칸을 늘려가며, 최종적으로는 전체 구절을 직접 입력해 검증한다.

### 2-1. 3단계 테스트 흐름

| 단계 | 명칭 | 설명 | 빈칸 비율 |
|---|---|---|---|
| 1단계 | 빈칸 맛보기 | 본문 단어 중 글자 수가 긴 단어부터 우선 선정해 빈칸 처리 | 약 25% |
| 2단계 | 빈칸 늘리기 | 같은 방식으로 빈칸 비율을 확대 | 약 65% |
| 3단계 | 전체 암송 | 출처만 보여주고 전체 단어를 빈칸 처리 | 100% |

> 💡 1단계를 통과해야 2단계로 진입, 2단계 통과 후 3단계로 진입하는 순차 구조. (자신 있는 사용자를 위한 "3단계부터 시작" 옵션은 추후 검토)

### 2-2. 빈칸 선정 로직 (구현 확정)

- ~~운영자가 "암송" 열에 미리 지정한 빈칸 패턴을 사용~~ → **변경됨**: "암송" 열은 본문 전체와 토큰 수가 일치하지 않아(목록 화면용 짧은 미리보기였음) 채점 로직에는 사용하지 않기로 결정
- 대신 코드가 본문(`성경말씀`)을 공백 기준으로 토큰화하고, **글자 수가 긴 단어부터 우선** 빈칸 후보로 선정 (`pickBlankIndices` 함수)
- 빈칸 글자 수는 정답 글자 수와 동일한 칸 수로 표시 (예: 3글자 단어 → input 칸 3개)
- **알려진 한계**: 글자 길이만 기준으로 삼다 보니 2글자 조사("에서", "으로" 등)가 짧은 핵심 명사보다 먼저 빈칸으로 잡히는 경우가 있음 → 조사 제외 목록을 추가해 후순위로 미루는 보정이 향후 과제로 남음 (9장 참고)

### 2-3. 입력 방식 (구현 확정, v1.2에서 변경)

- ~~글자 단위 `<input maxlength="1">`을 빈칸 수만큼 나열~~ → **변경됨**: 모바일 한글 IME 조합 시 1글자 input이 자모 단위로 끊기는 문제가 있어, **빈칸 1개당 단어 단위 input 1개**로 확정. 정답 글자 수에 맞춰 input 너비를 잡아 길이 힌트를 준다 (`renderTestScreen`)
- **자동 진행 채점**: 입력값이 정답 단어와 같아지는 즉시 비교 (한글 조합 중이라도 정답과 일치하면 바로 통과)
  - 맞으면: 초록색 표시 + 입력 잠금(disabled) + 자동으로 다음 빈칸 포커스 이동
  - 틀리면: 빨간색 + 흔들림 애니메이션 → 약 0.4초 후 자동 초기화. 한글 IME 조합 버퍼가 남지 않도록 `blur()`로 세션을 끝낸 뒤 비우고 재포커스
  - 모든 빈칸을 다 맞히면 자동으로 다음 단계 버튼 표시 (제출 버튼 없음)
- **정답 보기**: 빈칸 단어를 강조 표시하는 '정답 보기/돌아가기' 토글 제공 (입력 내용은 유지)
- **음성 암송**: 1차 범위에서 제외 예정이었으나 **구현 완료** (2-4 참고)
- 힌트 기능(한 글자 보기)은 1차 범위에서 제외

---

### 2-4. 음성 암송 (구현 완료, 부록→본문 승격)

원래 부록의 향후 확장 아이디어였으나 1차 버전에 구현 완료.

- 각 테스트 화면에 **🎤 암송 시작/■ 암송 종료 토글 버튼** 제공 (Web Speech API `SpeechRecognition`, `lang=ko-KR`)
- '암송 종료'를 누를 때까지 계속 듣기 — 브라우저가 침묵(no-speech) 등으로 세션을 끊어도 자동 재시작
- 말한 내용과 본문을 **LCS(최장 공통 부분수열)** 로 정렬해 단어 일치율(%) 계산. 맞은 단어/틀린 단어를 색으로 표시
- **통과 기준: 일치율 90% 이상 → 암송 완료(3단계)로 저장** (`VOICE_PASS = 90`)
- **예외 처리**: 카카오톡 인앱 브라우저(마이크 차단)는 '외부 브라우저로 열기' 안내, 음성인식 미지원 브라우저는 타이핑 안내

## 3. 화면 흐름 (User Flow)

등록 절차 없이 누구나 바로 테스트를 시작할 수 있도록 진입 장벽을 최소화한다.

```
[목록 화면]
  └─ 26개 구절 카드 (회차 + 출처 + "암송" 열 미리보기 + 진행 상태 배지)
       └─ 진행 배지: 미시도 / 1단계 완료 / 2단계 완료 / 암송 완료 🙌 (localStorage 기반)
       ↓ 카드 클릭 (이미 통과한 단계가 있으면 그 다음 단계부터 이어서 시작)
[테스트 화면 — 1단계]
  └─ 단계 배지 + 출처 → 단어 단위 빈칸 입력창 → 입력 시 자동 채점
  └─ 보조: 설교 영상 배너 · 정답 보기 토글 · 🎤 음성 암송
       ↓ 전체 정답 시 자동 통과
[결과 영역 — 단계 내 인라인 표시]
  └─ 통과 시 "2단계로" 버튼 / 부분 실패는 발생하지 않음(자동 교정 방식)
       ↓
[테스트 화면 — 2단계] → [테스트 화면 — 3단계]
       ↓ 3단계까지 통과 (또는 음성 암송 90% 이상)
[완료 표시 — 인라인 구현 완료]
  └─ "암송 완료 🙌" 배지 + 설교제목/영상 URL 배너 + 목록 진행 배지 자동 갱신
  └─ (별도 전체화면 완료 페이지 + QR 연계는 향후 과제)
```

> 💡 원래 기획은 "정확도(%) 기반 통과/실패 판정"이었으나, 구현 과정에서 "한 글자씩 자동 교정하며 끝까지 다 맞춰야 통과"하는 방식으로 더 명확하게 확정됨. 따라서 결과 화면은 실패가 없고, 항상 "완료 → 다음 단계"로 흐른다.

## 4. 데이터 구조 (Google Sheets)

### 4-1. 시트1 — 말씀 마스터 데이터 (구현 확정, 기획서 초안과 다름)

실제 운영 중인 헤더 구조:

| 열 | 헤더명 | 설명 | 예시 |
|---|---|---|---|
| A | No | 회차 번호 | 1 |
| B | 일자 | 주차 날짜 | 2026-01-04 |
| C | 말씀1 | 출처 전체 표기 | 시편 119편 105절 |
| D | 말씀2 | 출처 약식 표기 | 시 119:105 |
| E | 성경말씀 | 본문 전체 (채점 기준 원문) | 주의 말씀은 내 발의 등이요... |
| F | 설교제목 | 관련 설교 제목 | 오직 성경, 말씀이 답이다! |
| G | 목사님 | 설교자 | 차동혁 담임목사 |
| H | 암송 | 목록 화면 미리보기용 (일부만 가린 짧은 힌트) | 주의 OOO O |
| I | URL | 설교 영상 링크 | https://... |

> ⚠️ **"암송" 열(H) 사용 범위 주의**: 본문 전체와 토큰 수가 일치하지 않으므로 빈칸 생성·채점 로직에는 사용하지 않는다. 목록 화면에서 카드에 살짝 보여주는 미리보기 용도로만 사용.

### 4-2. 시트2 — 진행 기록 (아직 미구현, 선택 기능)

| 열 | 헤더명 | 설명 |
|---|---|---|
| A | 일시 | 제출 시각 |
| B | 식별자 | 이름/닉네임 (선택 입력) |
| C | 구절No | 테스트한 구절 번호 |
| D | 단계 | 통과한 단계 |
| E | 정확도 | (자동 교정 방식이므로 항상 100%에 가까움 — 통과 여부로 단순화 검토) |

> 💡 1차 버전은 시트2 없이 브라우저 `localStorage`만으로 진행 상태를 관리하는 것으로 확정·구현되었다. 키 `memorize-progress`, 값 `{ "1": { stage: 2, passed: true }, ... }`(stage = 통과한 최고 단계 0~3). 더 높은 단계 통과 시에만 갱신해 복습으로 후퇴하지 않는다.

## 5. 시스템 아키텍처

기존 완주 등록 시스템과 동일한 무료 구성(GitHub Pages + Apps Script + Google Sheets)을 재사용하되, 별도 프로젝트로 분리한다.

| 영역 | 역할 | 기술 | 상태 |
|---|---|---|---|
| 프론트엔드 | 테스트 UI 렌더링, 빈칸 입력 처리, 채점, 음성 암송, 결과 표시 | Vanilla JS · GitHub Pages | **배포 완료** (GitHub Pages 공개) |
| 데이터 캐시 | 말씀 데이터 정적 캐시(1순위) | `verses.json` (저장소 내 파일) | **운영 중** — GitHub Actions가 주기적으로 갱신 |
| 데이터 API | 말씀 마스터 데이터 조회(GET, 폴백) | Google Apps Script (Web App) | **배포 완료** — `verses.json` 실패 시 폴백 |
| 데이터 저장 | 말씀 마스터 데이터 | Google Sheets | **데이터 입력 완료** |
| 진행 상태 | 구절별 통과 단계 저장 | 브라우저 `localStorage` | **구현 완료** (`memorize-progress`) |

### 5-1. 배포된 API

```
GET https://script.google.com/macros/s/AKfycbzO4GDAy0hJBbZ-L3hVuZQI4cqnjiZdy2afUujnxmmAr8NAh1lJURhrfT37PaFanPR4PA/exec
```

응답 형식:
```json
{
  "verses": [
    {
      "no": 1,
      "date": "2026-01-04",
      "refFull": "시편 119편 105절",
      "refShort": "시 119:105",
      "text": "주의 말씀은 내 발의 등이요 내 길에 빛이니이다",
      "sermonTitle": "오직 성경, 말씀이 답이다!",
      "pastor": "차동혁 담임목사",
      "hintText": "주의 OOO O",
      "url": "https://..."
    }
  ]
}
```

액세스 권한: 모든 사용자. 26개 행 전체 정상 반환 확인됨.

### 5-2. 데이터 흐름 (v1.2 갱신)

```
1) 페이지 로드
   → 1순위: 저장소의 정적 verses.json 읽기 (빠르고 안정적)
   → 실패 시 2순위: Apps Script Web App에 GET 폴백
   → 받은 JSON으로 26개 구절 카드 렌더링 (+ localStorage 진행 배지 표시)

   ※ verses.json은 GitHub Actions 워크플로(.github/workflows/refresh-verses.yml)가
     주기적으로 시트 API를 호출해 자동 갱신 → 운영팀이 시트만 고치면 사이트에 반영됨

2) 사용자가 빈칸에 단어 입력
   → 채점은 100% 프론트엔드(JS)에서 즉시 처리 (서버 왕복 없음)
   → 통과 단계는 브라우저 localStorage("memorize-progress")에 저장 (서버 저장 없음)
```

## 6. 채점 로직

### 6-1. 비교 규칙 (구현 확정)

- 글자 단위(음절) 1:1 비교 — 입력 즉시 비교, 지연 없음
- 맞은 글자는 즉시 잠금(disabled) 처리되어 재수정 불가
- 틀린 글자는 0.3초간 빨간색으로 보여준 뒤 자동으로 지워져 재입력 유도
- 전체 빈칸을 다 맞혀야 해당 단계 완료 — 부분 정확도 개념 없음 (기존 "정확도 80%" 기준은 자동교정 방식 도입으로 의미가 약해짐, 9장 참고)

### 6-2. 원래 기준이었던 단계 통과 기준 (참고용, 현재 구현과 다름)

| 단계 | 원래 기준 | 현재 구현 |
|---|---|---|
| 1단계 | 정확도 80% 이상 | 전체 빈칸 완전 정답 시 통과 |
| 2단계 | 정확도 80% 이상 | 전체 빈칸 완전 정답 시 통과 |
| 3단계 | 정확도 90% 이상 | 전체 빈칸 완전 정답 시 통과 |

> 💡 자동교정 방식에서는 "오답 횟수"나 "재시도 횟수"를 기록해 별도의 숙련도 지표로 쓰는 방향이 더 적합할 수 있다. 예: 한 번도 안 틀리고 통과 = ⭐⭐⭐, 몇 번 틀리고 통과 = ⭐⭐ 같은 식. 9장 향후 과제로 남김.

## 7. 화면별 상세 기획

### 7-1. 목록 화면 (구현 완료)

- 26개 구절을 카드 그리드로 나열(최신 주차 → 1주차 순), 클릭 시 테스트 화면 진입
- 진행 상태 배지(미시도 / 1·2단계 완료 / 암송 완료 🙌) **구현 완료** — localStorage 기반
- 상단에 '🔔 매일 암송 구절 알림 받기'(reminders.html) 링크 제공

### 7-2. 테스트 화면 (구현 완료)

- 상단: 현재 단계 배지 + 출처(약식) + '← 목록' 버튼
- 중앙: 고정 단어 + **단어 단위 빈칸 입력창**(글자 수에 맞춘 너비)
- 보조 UI: 설교 영상 배너 · '정답 보기' 토글 · '🎤 암송 시작' 음성 암송
- 자동 채점 — 별도 제출 버튼 없음
- 이미 통과한 단계가 있으면 그 다음 단계부터 이어서 시작
- 완료 시 다음 단계 버튼 인라인 표시

### 7-3. 결과 표시 (구현 완료, 기획 대비 단순화됨)

- 정확도(%) 큰 숫자 표시 + 통과 라벨
- 오답 하이라이트는 "그 즉시 보여주고 지우는" 방식으로 대체되어, 별도의 "틀린 부분 모아보기" 화면은 없음

### 7-4. 완료 표시 (3단계까지 통과) — **인라인 구현 완료**

- 3단계 통과 시 결과 영역에 "암송 완료 🙌" 배지 표시 + 목록 진행 배지 자동 갱신
- 테스트 화면 내 설교 영상 배너로 설교 제목·URL 연결 (소책자 연계 정보 재사용)
- **향후 과제**: 별도 전체화면 완료 페이지 + QR 연계, "다음 구절로 이동" 버튼은 미구현

## 8. 개인정보 보호 원칙

- 기본 모드는 완전 익명 — 이름, 연락처 등 어떤 개인정보도 입력받지 않음
- 진행 기록 저장(선택 기능, 미구현) 사용 시에도 휴대폰 번호 등 민감 정보는 수집하지 않으며, 가벼운 닉네임 또는 이름 정도만 선택적으로 입력받음
- 수집되는 데이터는 "구절 번호 + 단계 + 통과 여부" 수준으로 최소화
- 기존 완주 등록 시스템의 식별자(교구+목장+성명)와는 별개로 관리하여 두 데이터베이스가 서로 연결되지 않도록 함
